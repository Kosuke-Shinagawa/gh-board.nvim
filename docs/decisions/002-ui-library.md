# 002: UI ライブラリに nui.nvim を採用する

> 日付: 2026-06-13
> ステータス: 採用

## 背景

Kanban ボード・カード詳細・フォームを Neovim のフロートウィンドウで構築するために UI ライブラリが必要。
Neovim の `vim.api.nvim_open_win()` を直接使うこともできるが、レイアウト計算・キーマップ管理・フォーカス制御を自前実装すると保守コストが高くなる。

## 判断

**nui.nvim を採用する。**

## 理由

| 選択肢 | 評価 |
|--------|------|
| `nui.nvim`（採用） | Popup / Layout / Input / Menu の 4 コンポーネントを提供。neo-tree.nvim・noice.nvim 等の有名プラグインが採用しており実績十分。API が安定しておりドキュメントも充実 |
| `nvim-window-picker` + 自前実装 | フロートウィンドウ管理のみで、フォームやメニューは自前実装が必要。開発コストが高い |
| `telescope.nvim` の内部 API | picker 以外への流用は設計外であり、フォームや Kanban 描画には向かない |
| 完全自前実装（`nvim_open_win` 直接） | 依存ゼロにできるが、フォーカス管理・リサイズ・ESC ハンドリングなどの実装で数百行かかる。バグのリスクも高い |

nui.nvim は必須依存として追加するが、広く使われており学習コストは低い。

## 結果・トレードオフ

- **メリット**: Popup（詳細表示）・Layout（Kanban ボード）・Input（フォーム）・Menu（カラム選択）をコンポーネントとして使い分けられる。フォーカス制御・ESC ハンドリングが組み込み済み
- **デメリット**: nui.nvim が必須依存になる。過去に breaking change の実績があるためバージョンをピン留めする必要あり
- **対策**: lazy.nvim の設定例に `commit` または `tag` によるピン留めを記載する
