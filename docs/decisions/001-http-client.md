# 001: HTTP クライアントに plenary.curl を採用する

> 日付: 2026-06-13
> ステータス: 採用

## 背景

GitHub GraphQL API への HTTP リクエストを非同期で行う手段が必要。
Neovim には組み込みの HTTP クライアントがないため、外部ライブラリまたはシステムコマンドを使う必要がある。

## 判断

**plenary.nvim の `plenary.curl` モジュールを採用する。**

## 理由

| 選択肢 | 評価 |
|--------|------|
| `plenary.curl`（採用） | Neovim エコシステムで最も普及している HTTP ラッパー。telescope.nvim・lazy.nvim 等が依存しており、ユーザーの環境にすでに入っていることが多い。非同期 (`job`) と同期両対応。GraphQL POST リクエストも自然に書ける |
| `vim.fn.jobstart` + curl コマンド | plenary なしで動くが、JSON のエスケープ・レスポンスパースを自前で実装する必要があり保守コストが高い |
| `luasocket` | 外部 Lua ライブラリで Neovim 環境への導入が煩雑。Neovim コミュニティでの実績が薄い |
| `vim.loop`（libuv 直接） | 最も低レベルで依存ゼロだが、HTTP/HTTPS の実装を一から書く必要があり現実的でない |

plenary.nvim はすでに必須依存（テストフレームワークとしても使用）であるため、HTTP クライアントとして使っても依存数は増えない。

## 結果・トレードオフ

- **メリット**: 実装量が少ない。認証ヘッダーの付与・JSON エンコード・レスポンスパースが簡潔に書ける
- **デメリット**: plenary.nvim が必須依存になる。ただし広く普及しているため受け入れコストは低い
- **将来の移行パス**: Neovim 0.10+ の `vim.system()` + 外部 curl で plenary なしの実装に移行可能
